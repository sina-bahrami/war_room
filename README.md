# Prompcorp Tender Intelligence

This repository has two parts that work together:

- root-level data preparation scripts and generated files
- the deployable dashboard stack under `combine/`

The root scripts prepare procurement data from Tenders.Net HTML snapshots, AusTender, NSW eTendering, and Victorian CKAN datasets. The `combine/` application then ingests those files into PostgreSQL and serves the dashboard through a FastAPI backend and React frontend.

## Repository Layout

```text
tender/
|-- build_warroom_dashboard_data.py
|-- tenders_ingest.py
|-- vic_tenders_ckan.py
|-- *.html
|-- warroom_dashboard_data.json
|-- austender.jsonl
|-- vic_data/
`-- combine/
    |-- backend/
    |-- frontend/
    |-- k8s/
    |-- docker-compose.yml
    |-- DEPLOYMENT.md
    `-- DEPLOYMENT_K8S.md
```

## How The Data Flows

1. Tenders.Net HTML snapshots at the repo root are converted into `warroom_dashboard_data.json` by `build_warroom_dashboard_data.py`.
2. AusTender and optional NSW eTendering exports are generated by `tenders_ingest.py`.
3. Victorian procurement datasets are downloaded into `vic_data/` by `vic_tenders_ckan.py`.
4. The `combine/` ingestion worker reads those files from `/project-root` inside the containers or pods.

Important runtime rule:

- if `WARROOM_JSON_SOURCE_URL` is set, `combine/` runs in unified JSON mode and only ingests `warroom_dashboard_data.json`
- when unified JSON mode is enabled, the legacy source variables such as `AUSTENDER_SOURCE_URL` and `VIC_TENDERS_SOURCE_URL` are ignored

## Main Components

- `build_warroom_dashboard_data.py`: builds a unified dashboard seed file from `*Prompcorp _ Tenders.Net.html`
- `tenders_ingest.py`: fetches AusTender RSS data and NSW eTendering data and writes JSONL exports
- `vic_tenders_ckan.py`: downloads procurement-related CKAN datasets from DataVic into `vic_data/`
- `combine/backend`: FastAPI API, ingestion worker, normalized tender schema, database access
- `combine/frontend`: React/Vite dashboard
- `combine/k8s`: Kubernetes manifests for MicroK8s-style deployment

## Step 1: Build The Data Files

### 1. Build `warroom_dashboard_data.json` from Tenders.Net HTML

`build_warroom_dashboard_data.py` automatically scans the repo root for files matching:

- `*Prompcorp _ Tenders.Net.html`

It writes a unified snapshot file named `warroom_dashboard_data.json` by default.

Examples:

```bash
python build_warroom_dashboard_data.py
python build_warroom_dashboard_data.py --output warroom_dashboard_data.json
python build_warroom_dashboard_data.py --as-of-date 2026-06-01
```

Output:

- `warroom_dashboard_data.json`

### 2. Build `austender.jsonl` and optional `nsw.jsonl`

`tenders_ingest.py` writes JSONL exports. Those files can be consumed directly by `combine/`.

Examples:

```bash
python tenders_ingest.py austender --output austender.jsonl
python tenders_ingest.py nsw --output nsw.jsonl --modified-from 01-Jan-2026
python tenders_ingest.py all --austender-output austender.jsonl --nsw-output nsw.jsonl --modified-from 01-Jan-2026
```

Notes:

- `austender` fetches current AusTender ATM records from the RSS feed
- `nsw` fetches NSW eTendering records and can filter by `--modified-from`, `--keyword`, and `--rft-type`
- the live export format is JSONL, so use file names such as `austender.jsonl` and `nsw.jsonl`

### 3. Build `vic_data/` from DataVic CKAN

`vic_tenders_ckan.py` writes one directory per dataset, along with `metadata.json`, `manifest.json`, and extracted resource payloads.

Examples:

```bash
python vic_tenders_ckan.py search --output-dir vic_data --rows 10
python vic_tenders_ckan.py dataset --id dh-annual-report-2023-24-procurement-including-ict-and-advertising --output-dir vic_data
```

Output:

- `vic_data/search_results.json`
- `vic_data/skipped_datasets.json`
- `vic_data/<dataset-slug>/metadata.json`
- `vic_data/<dataset-slug>/manifest.json`
- `vic_data/<dataset-slug>/*.json`

### 4. Optional internal pipeline file

If you want to load internal bid pipeline data in legacy multi-source mode, point `PROMPCORP_PIPELINE_SOURCE_URL` to a CSV or JSON file shaped like the sample in:

- `combine/backend/ingestion/samples/prompcorp_pipeline.csv`

## Step 2: Choose The Ingestion Mode

### Recommended: unified JSON mode

Use this when `warroom_dashboard_data.json` is your main curated dataset.

Set:

```env
ENABLE_SAMPLE_DATA=false
WARROOM_JSON_SOURCE_URL=/project-root/warroom_dashboard_data.json
```

Leave the legacy source variables blank.

This is the cleanest path for the current root-level workflow because `build_warroom_dashboard_data.py` already produces the unified dashboard seed file expected by the `warroom_unified_json` connector.

### Legacy multi-source mode

Use this when you want `combine/` to ingest split source exports such as AusTender, VIC CKAN, NSW, QLD, and internal pipeline data.

Leave `WARROOM_JSON_SOURCE_URL` blank and set one or more of:

```env
ENABLE_SAMPLE_DATA=false
WARROOM_JSON_SOURCE_URL=
AUSTENDER_SOURCE_URL=/project-root/austender.jsonl
NSW_ETENDERING_SOURCE_URL=/project-root/nsw.jsonl
VIC_TENDERS_SOURCE_URL=/project-root/vic_data
QLD_PROCUREMENT_SOURCE_URL=
PROMPCORP_PIPELINE_SOURCE_URL=/project-root/prompcorp_pipeline.csv
```

Supported source locations:

- local filesystem paths
- `file://` paths
- remote `http://` or `https://` URLs

## Step 3: Deploy With Docker Compose

This is the fastest way to run the whole project locally or on a single VM.

### 1. Create the environment file

Copy `combine/.env.example` to `combine/.env`, then edit it.

Recommended unified JSON configuration:

```env
PROJECT_NAME=Prompcorp Tender Intelligence
POSTGRES_USER=prompcorp
POSTGRES_PASSWORD=change-me
POSTGRES_DB=tender_intelligence
POSTGRES_HOST=postgres
POSTGRES_PORT=5432
REDIS_HOST=redis
REDIS_PORT=6379
ALLOWED_ORIGINS=http://localhost:8080,http://127.0.0.1:8080
INGESTION_INTERVAL_SECONDS=900
ENABLE_SAMPLE_DATA=false
WARROOM_JSON_SOURCE_URL=/project-root/warroom_dashboard_data.json
AUSTENDER_SOURCE_URL=
NSW_ETENDERING_SOURCE_URL=
VIC_TENDERS_SOURCE_URL=
QLD_PROCUREMENT_SOURCE_URL=
PROMPCORP_PIPELINE_SOURCE_URL=
VITE_API_BASE_URL=/api
```

Recommended legacy multi-source example:

```env
ENABLE_SAMPLE_DATA=false
WARROOM_JSON_SOURCE_URL=
AUSTENDER_SOURCE_URL=/project-root/austender.jsonl
NSW_ETENDERING_SOURCE_URL=/project-root/nsw.jsonl
VIC_TENDERS_SOURCE_URL=/project-root/vic_data
PROMPCORP_PIPELINE_SOURCE_URL=/project-root/prompcorp_pipeline.csv
```

Important path note:

- `combine/docker-compose.yml` mounts the repository root into the containers as `/project-root`
- your generated files must exist in this repository for those paths to resolve

### 2. Start the stack

Run these commands from `combine/`:

```bash
docker compose up -d --build
```

Services started:

- `postgres`
- `redis`
- `backend`
- `ingestor`
- `frontend`

### 3. Trigger the first ingestion

The ingestor runs continuously on its configured interval, but it is usually better to force one immediate load after deployment or after replacing data files.

From `combine/`:

```bash
docker compose exec ingestor python -m ingestion.main --once
```

Or trigger it through the API:

```bash
curl -X POST http://localhost:8080/api/tenders/admin/run-ingestion
```

### 4. Verify the deployment

Open:

- `http://localhost:8080`
- `http://localhost:8080/health`
- `http://localhost:8080/api/dashboard/summary`
- `http://localhost:8080/api/sources/health`

Useful commands:

```bash
docker compose ps
docker compose logs backend
docker compose logs ingestor
docker compose logs frontend
```

## Step 4: Deploy With Kubernetes

The Kubernetes manifests under `combine/k8s/` target a self-hosted MicroK8s-style environment.

Detailed runbook:

- `combine/DEPLOYMENT_K8S.md`

### 1. Enable MicroK8s addons

```bash
microk8s enable dns
microk8s enable storage
microk8s enable ingress
```

### 2. Build the images

From the repository root:

```bash
docker build -t prompcorp-backend:local ./combine/backend
docker build -t prompcorp-frontend:local ./combine/frontend
```

### 3. Import the images into MicroK8s

```bash
docker save prompcorp-backend:local | microk8s ctr image import -
docker save prompcorp-frontend:local | microk8s ctr image import -
```

### 4. Copy the data files to the Kubernetes host path

The backend and ingestor deployments mount this host directory:

- `/var/snap/microk8s/common/prompcorp-tender-intelligence`

Inside the pods it appears as:

- `/project-root`

For unified JSON mode:

```bash
sudo mkdir -p /var/snap/microk8s/common/prompcorp-tender-intelligence
sudo cp warroom_dashboard_data.json /var/snap/microk8s/common/prompcorp-tender-intelligence/
```

For legacy multi-source mode, copy the files and directories you plan to reference:

```bash
sudo mkdir -p /var/snap/microk8s/common/prompcorp-tender-intelligence
sudo cp austender.jsonl /var/snap/microk8s/common/prompcorp-tender-intelligence/
sudo cp nsw.jsonl /var/snap/microk8s/common/prompcorp-tender-intelligence/
sudo cp -r vic_data /var/snap/microk8s/common/prompcorp-tender-intelligence/
```

### 5. Set the Kubernetes config

Edit `combine/k8s/configmap.yaml`.

Recommended unified JSON values:

```yaml
ENABLE_SAMPLE_DATA: "false"
WARROOM_JSON_SOURCE_URL: "/project-root/warroom_dashboard_data.json"
AUSTENDER_SOURCE_URL: ""
NSW_ETENDERING_SOURCE_URL: ""
VIC_TENDERS_SOURCE_URL: ""
QLD_PROCUREMENT_SOURCE_URL: ""
PROMPCORP_PIPELINE_SOURCE_URL: ""
```

Recommended legacy multi-source values:

```yaml
ENABLE_SAMPLE_DATA: "false"
WARROOM_JSON_SOURCE_URL: ""
AUSTENDER_SOURCE_URL: "/project-root/austender.jsonl"
NSW_ETENDERING_SOURCE_URL: "/project-root/nsw.jsonl"
VIC_TENDERS_SOURCE_URL: "/project-root/vic_data"
PROMPCORP_PIPELINE_SOURCE_URL: "/project-root/prompcorp_pipeline.csv"
```

### 6. Create the secret

Copy `combine/k8s/secret.example.yaml` to `combine/k8s/secret.yaml`, set a real `POSTGRES_PASSWORD`, then apply the namespace and secret:

```bash
microk8s kubectl apply -f combine/k8s/namespace.yaml
microk8s kubectl apply -f combine/k8s/secret.yaml
```

### 7. Deploy the stack

```bash
microk8s kubectl apply -k combine/k8s
```

### 8. Run one ingestion cycle

```bash
microk8s kubectl apply -f combine/k8s/ingestion-job.yaml
microk8s kubectl logs -n prompcorp-tender-intelligence job/ingestor-once
```

### 9. Verify the deployment

```bash
microk8s kubectl get pods -n prompcorp-tender-intelligence
microk8s kubectl get svc -n prompcorp-tender-intelligence
microk8s kubectl get ingress -n prompcorp-tender-intelligence
```

Access the dashboard:

- NodePort: `http://<microk8s-node-ip>:32080`
- Ingress: `http://prompcorp-tenders.local`

## Operational Notes

- keep `ENABLE_SAMPLE_DATA=false` when using real exports
- rerun ingestion after replacing `warroom_dashboard_data.json`, `austender.jsonl`, `vic_data/`, or any other source file
- in unified JSON mode, the ingestor replaces the tender dataset from `warroom_dashboard_data.json`
- in legacy mode, the ingestor upserts records from each enabled source connector

## Troubleshooting

- If the dashboard shows sample data, check `ENABLE_SAMPLE_DATA` and make sure your source variables are set correctly.
- If real source files are being ignored, confirm whether `WARROOM_JSON_SOURCE_URL` is still set. When it is set, the legacy source connectors do not run.
- If Kubernetes pods cannot see your data, make sure the files exist under `/var/snap/microk8s/common/prompcorp-tender-intelligence` on the host.
- If `vic_data/` was copied to Kubernetes, preserve the directory structure because the VIC connector expects dataset subdirectories with `metadata.json` and `manifest.json`.
